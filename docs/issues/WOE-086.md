# WOE-086: Decompose Telemetry HUD Painter Into Per-Element Painters

**ID:** WOE-086
**Phase:** Quality -- Refactoring
**Depends on:** WOE-070, WOE-071, WOE-072, WOE-073, WOE-074

---

## Description

`HudPainter` in `lib/shared/telemetry_hud.dart` is a single ~600-line `CustomPainter` whose `paint()` renders the reticle (WOE-070), heading tape (WOE-071), pitch ladder (WOE-072), roll indicator (WOE-073), and data strip (WOE-074). Modifying or testing one element means reasoning about all five.

Split into one painter class per HUD element, each independently testable, composed either by sequential delegation inside a thin `HudPainter` or by stacked `CustomPaint` widgets (pick whichever avoids extra saveLayer/composite cost — measure if unsure).

Pure refactor: pixel output must be unchanged.

---

## Definition of Done

- [ ] One painter class per element: `ReticlePainter`, `HeadingTapePainter`, `PitchLadderPainter`, `RollIndicatorPainter`, `DataStripPainter`
- [ ] Each painter takes only the data it needs (no shared god-state object beyond `HudData` slices)
- [ ] Shared text/shadow paint helpers extracted once (coordinate with WOE-087's debug-overlay text utilities — same helper, one home)
- [ ] `shouldRepaint` implemented per painter so unchanged elements skip repaints
- [ ] Unit tests: at minimum one golden or paint-recording test per painter verifying it draws at expected positions for a known `HudData`
- [ ] No visual change on device (manual spot check in AR view)

---

## Sub-tasks

- [ ] Identify the data dependencies of each `paint()` region
- [ ] Extract shared text-rendering helpers into `lib/shared/hud_paint_utils.dart`
- [ ] Move each region into its own painter class + file under `lib/shared/hud/`
- [ ] Wire composition; keep `TelemetryHud` widget API unchanged
- [ ] Add painter tests; run full suite
