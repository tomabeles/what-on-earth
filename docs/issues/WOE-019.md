# WOE-019: Wire orientationStream to CesiumJS Camera

**ID:** WOE-019
**Phase:** 3 — Sensor Fusion and AR Camera
**Step:** 3.3
**Depends on:** WOE-018, WOE-014

---

## Description

Connect `SensorFusionEngine.orientationStream` to the bridge so that every `DeviceOrientation` update is sent to CesiumJS as an `UPDATE_ORIENTATION` message. CesiumJS receives it and calls `camera.setView()` with the new heading/pitch/roll, keeping the position from the last `UPDATE_POSITION` message.

This is the step where moving the device physically changes what the globe shows — the core AR interaction.

**Reference:** TECH_SPEC §7.2, §8.1, §3.4

---

## Definition of Done

- [ ] Every `DeviceOrientation` from the engine triggers an `UPDATE_ORIENTATION` bridge message
- [ ] CesiumJS camera heading/pitch/roll updates at ≥ 50 Hz (sensor cadence)
- [ ] Globe orientation changes visibly when the device is rotated
- [ ] `UPDATE_POSITION` and `UPDATE_ORIENTATION` are composed correctly: position sets camera destination, orientation sets camera heading/pitch/roll independently
- [ ] When `reliable: false` is set on the orientation, the bridge still sends the message but includes the flag (UI banner is added in WOE-021)

---

## Sub-tasks

- [ ] In `GlobeView`, subscribe to `orientationStreamProvider` using `ref.listen`
- [ ] On each `DeviceOrientation`, call `bridgeController.send(OutboundMessage.updateOrientation, orientation.toJson())`
- [ ] In CesiumJS `main.js`, implement the `UPDATE_ORIENTATION` handler:
  ```javascript
  handlers['UPDATE_ORIENTATION'] = (payload) => {
    const currentPos = viewer.camera.position;
    viewer.camera.setView({
      destination: currentPos,  // preserve position
      orientation: {
        heading: Cesium.Math.toRadians(payload.headingDeg),
        pitch:   Cesium.Math.toRadians(payload.pitchDeg),
        roll:    Cesium.Math.toRadians(payload.rollDeg),
      },
    });
  };
  ```
- [ ] Verify the globe rotates correctly when the device is physically rotated (manual test on device)
- [ ] Profile: verify the JS bridge call latency does not cause UI jank; if needed, throttle to 30 Hz for the bridge while keeping the filter at 50 Hz
