# WOE-014: Wire PositionController to GlobeView

**ID:** WOE-014
**Phase:** 2 — Position Module
**Step:** 2.7
**Depends on:** WOE-013, WOE-004

---

## Description

Connect the `PositionController` to the `GlobeView` so that every position update drives a `UPDATE_POSITION` bridge message to CesiumJS, moving the globe camera to the correct ISS location. This is the step where the globe first moves in real time with the ISS.

**Reference:** TECH_SPEC §8.1, §7.1

---

## Definition of Done

- [ ] When `GLOBE_READY` is received, the `GlobeView` starts listening to `PositionController.positionStream`
- [ ] Each `OrbitalPosition` is immediately sent to CesiumJS as an `UPDATE_POSITION` bridge message
- [ ] CesiumJS `UPDATE_POSITION` handler calls `viewer.camera.setView({ destination: Cesium.Cartesian3.fromDegrees(lon, lat, altMeters) })` — note: `altKm * 1000` to convert to meters
- [ ] The globe camera visibly moves as the ISS orbits
- [ ] The `source` field from `OrbitalPosition` is included in the bridge payload (used later by the status indicator)
- [ ] No position updates are sent before `GLOBE_READY` is received

---

## Sub-tasks

- [ ] In `GlobeView` (or its corresponding controller), use `ref.listen(positionStreamProvider, ...)` to react to each new `OrbitalPosition`
- [ ] On each update, call `bridgeController.send(OutboundMessage.updatePosition, position.toJson())`
- [ ] In CesiumJS `main.js`, implement the `UPDATE_POSITION` handler:
  ```javascript
  handlers['UPDATE_POSITION'] = (payload) => {
    viewer.camera.setView({
      destination: Cesium.Cartesian3.fromDegrees(payload.lon, payload.lat, payload.altKm * 1000),
      orientation: { heading: 0, pitch: Cesium.Math.toRadians(-90), roll: 0 },
    });
  };
  ```
- [ ] Test on device: open the app, wait ~60 seconds, observe that the globe moves
- [ ] Optionally: draw a small crosshair entity at the ISS nadir point for visual confirmation
