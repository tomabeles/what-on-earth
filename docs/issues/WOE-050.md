# WOE-050: Performance Profiling and Optimization

**ID:** WOE-050
**Phase:** 7 — NFR Pass and Polish
**Step:** 7.1
**Depends on:** WOE-024, WOE-019

---

## Description

Profile the app's runtime performance on target devices and ensure it meets the PRD's non-functional performance targets: sensor fusion at 50 Hz, globe at ≥ 30 fps, cold launch ≤ 5 seconds.

**Reference:** PRD §7.1, TECH_SPEC §3.6

---

## Definition of Done

- [ ] Cold launch to AR view active ≤ 5 seconds on a mid-range device (tested on at least one mid-range Android and one iPhone)
- [ ] Sensor fusion `orientationStream` emitting at ≥ 45 Hz under normal conditions (target 50 Hz; 45 Hz acceptable due to Android delivery variability)
- [ ] CesiumJS globe frame rate ≥ 30 fps during active AR session on a mid-range device
- [ ] Bridge call overhead (Flutter → JS for `UPDATE_ORIENTATION`) does not cause measurable UI jank (no frame drops > 16 ms at the Flutter level)
- [ ] All performance bottlenecks identified are either resolved or documented as known limitations
- [ ] A brief performance test notes document is committed to `docs/` summarizing findings

---

## Sub-tasks

- [ ] Measure cold launch time: instrument `main.dart` with `DateTime.now()` at start and in `GLOBE_READY` handler; log the delta on first launch
- [ ] Profile sensor fusion isolate using Flutter DevTools CPU profiler; check for unnecessary allocations per tick
- [ ] Measure `orientationStream` delivery rate: add a counter in the Flutter sensor fusion consumer, print Hz every 5 seconds
- [ ] Measure globe FPS: the `FRAME_RATE` bridge message (added in WOE-024) reports CesiumJS FPS every 5 seconds; log it
- [ ] If bridge call frequency causes jank: throttle `UPDATE_ORIENTATION` from 50 Hz to 30 Hz by dropping intermediate frames (keep the filter running at 50 Hz for accuracy, but only send to JS at 30 Hz)
- [ ] If cold launch > 5 s: identify bottleneck (tile server start? Globe init? Supabase init?); consider parallelizing startup tasks
- [ ] If globe FPS < 30: profile CesiumJS with Chrome DevTools (via Android WebView remote debugging); common causes: too many GeoJSON entities, high-frequency camera updates
- [ ] Commit `docs/performance_notes.md` with test device specs, measured values, and any optimizations applied
