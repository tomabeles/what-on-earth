# WOE-053: Security Audit and Hardening

**ID:** WOE-053
**Phase:** 7 — NFR Pass and Polish
**Step:** 7.4
**Depends on:** WOE-044, WOE-045

---

## Description

Perform a security review of the app against the PRD's security requirements and fix any issues found. Key areas: TLS on all external calls, pin data encrypted at rest, minimal permissions, no unauthorized telemetry.

**Reference:** PRD §7.4 NFR-SEC-001 through NFR-SEC-005

---

## Definition of Done

- [ ] All external HTTP calls use HTTPS (verified by inspection and network proxy test)
- [ ] Pin data (name, note, coordinates) confirmed encrypted at rest via `flutter_secure_storage` or OS-provided encryption (drift uses SQLite which is encrypted by the device's file-system encryption on both iOS and Android by default in the secure enclave; confirm this is sufficient or add SQLCipher)
- [ ] `AndroidManifest.xml` contains only required permissions: `INTERNET`, `CAMERA`, `READ_EXTERNAL_STORAGE` (if needed for tile pre-seed), `ACCESS_NETWORK_STATE`
- [ ] `Info.plist` contains only required usage descriptions: `NSCameraUsageDescription`, `NSMotionUsageDescription`
- [ ] No analytics, crash reporting, or telemetry calls are made without explicit user opt-in (Sentry is opt-in, added in WOE-058)
- [ ] Supabase anon key is injected via `--dart-define` (not hardcoded in source)
- [ ] Security findings documented in `docs/security_review.md`

---

## Sub-tasks

- [ ] Audit all `dio` calls in the codebase: verify every `baseUrl` starts with `https://`; add a `dio` interceptor that throws in debug mode if an HTTP URL is used
- [ ] Verify `CelesTrak` TLE endpoint uses HTTPS: `https://celestrak.org/...` ✓
- [ ] Verify `wheretheiss.at` endpoint uses HTTPS: `https://api.wheretheiss.at/...` ✓
- [ ] Check local tile server: `http://localhost:8765` is acceptable (localhost, no sensitive data traverses it)
- [ ] Verify pin data encryption: drift uses the OS file system. On iOS, the database file is created in the app sandbox which is protected by Data Protection (requires passcode). On Android, the app files are in internal storage which is not accessible to other apps. Document this as the encryption mechanism. If the team requires explicit encryption: add `sqlcipher_flutter_libs` and configure SQLCipher.
- [ ] Review `AndroidManifest.xml`: remove any permissions added by packages that are not needed (common culprits: `READ_PHONE_STATE`, `VIBRATE` — remove if present and not needed)
- [ ] Search codebase for hardcoded secrets: `grep -r 'eyJ' lib/` (JWT pattern), `grep -r 'sk_' lib/` (Stripe-style keys); verify none found
- [ ] Add an interceptor assertion: in debug builds, assert all HTTP calls use HTTPS
- [ ] Write `docs/security_review.md` documenting each check and result
