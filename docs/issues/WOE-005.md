# WOE-005: Android WebView Transparency Validation and Fix

**ID:** WOE-005
**Phase:** 1 — Foundation
**Step:** 1.5
**Depends on:** WOE-004

---

## Description

Validate that the WebView is fully transparent on Android (no white flash on load, no opaque background at rest) and apply the documented workaround if needed. The `flutter_inappwebview` v6 `transparentBackground: true` setting has a known Android bug (Issue #99). The fix involves using TextureView rendering mode (`useHybridComposition: false`) and, if still present, a platform channel call to set `WebView.setBackgroundColor(0)` natively.

This is a validation + bugfix step, not feature work. It must be run on a physical Android device, not just an emulator, because Android WebView transparency behavior differs between emulator and hardware.

**Reference:** TECH_SPEC §3.3

---

## Definition of Done

- [ ] On a physical Android device (API 12+): WebView loads with no white flash
- [ ] On a physical Android device: globe background outside the Earth sphere is fully transparent (device wallpaper or a colored Flutter background behind the WebView is visible)
- [ ] On iOS: no regression introduced — globe still renders correctly
- [ ] Any workaround applied is documented with a comment referencing the upstream bug

---

## Sub-tasks

- [ ] On a physical Android device, load `GlobeView` and observe whether white flash or opaque background is present
- [ ] Confirm `useHybridComposition: false` is set in `InAppWebViewSettings` (TextureView mode — more reliable alpha blending than SurfaceView/HybridComposition on Android)
- [ ] If white flash persists: add a platform channel call in `onWebViewCreated` using `MethodChannel` to invoke `WebView.setBackgroundColor(Color.TRANSPARENT)` (Android-only, wrapped in `if (Platform.isAndroid)`)
- [ ] If opaque background persists after initial load: investigate whether the Cesium `scene.backgroundColor` setting is being applied before first render; if not, add a brief `postFrameCallback` delay before calling `setView`
- [ ] Place a solid-colored `Container` (e.g., deep blue) behind the `InAppWebView` in the widget tree; confirm the color is visible through the transparent areas of the globe
- [ ] Test on at least two Android API levels (API 31 and API 34) if devices are available
- [ ] Add an inline code comment in `globe_view.dart` referencing the bug: `// flutter_inappwebview#99: useHybridComposition: false required for reliable transparency on Android`
