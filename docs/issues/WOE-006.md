# WOE-006: Flutter↔JS Bridge Skeleton

**ID:** WOE-006
**Phase:** 1 — Foundation
**Step:** 1.6
**Depends on:** WOE-004

---

## Description

Implement the bidirectional message bridge between Flutter and CesiumJS as defined in TECH_SPEC §8. This issue creates the infrastructure — the message dispatch pattern, the handler registration, and the contract types — but only wires up two messages as proof of concept: `GLOBE_READY` (JS → Flutter, already added in WOE-004) and `UPDATE_POSITION` (Flutter → JS, sent with a hardcoded test position).

All other bridge messages are implemented in later issues. The goal here is to establish the pattern that all later issues follow, with typed message enums and a single `BridgeController` class that owns the `InAppWebViewController`.

**Reference:** TECH_SPEC §8, §6.1 (`lib/globe/bridge.dart`)

---

## Definition of Done

- [ ] `lib/globe/bridge.dart` exists with a `BridgeController` class that holds the `InAppWebViewController`
- [ ] All outbound message types are defined as a Dart enum or sealed class (even if only `UPDATE_POSITION` is implemented)
- [ ] All inbound message types are defined (even if only `GLOBE_READY` is handled)
- [ ] `BridgeController.send(MessageType, Map<String, dynamic> payload)` sends a message to the WebView
- [ ] CesiumJS `main.js` has a central `flutter_message` event listener that dispatches to per-type handler functions
- [ ] A hardcoded test `UPDATE_POSITION` is sent 3 seconds after `GLOBE_READY` is received; the CesiumJS handler logs the received position to the browser console
- [ ] Unit tests cover `BridgeController.send()` serialization

---

## Sub-tasks

- [ ] Create `lib/globe/bridge.dart`:
  - Define `enum OutboundMessage { updatePosition, updateOrientation, setTle, toggleLayer, syncPins, setMode, requestPassCalc }`
  - Define `enum InboundMessage { globeReady, mapTap, passCalcResult, frameRate }`
  - Implement `BridgeController` with a `WebViewController` reference
  - Implement `send(OutboundMessage type, Map<String, dynamic> payload)` using `controller.evaluateJavascript` with the `window.dispatchEvent(new CustomEvent('flutter_message', ...))` pattern from TECH_SPEC §8.1
  - Implement `registerHandlers(InAppWebViewController controller)` that calls `addJavaScriptHandler` for all inbound messages
- [ ] Update `web_globe/src/main.js`:
  - Add central `window.addEventListener('flutter_message', ...)` dispatcher
  - Add `handlers` object mapping message type strings to handler functions
  - Implement `UPDATE_POSITION` handler: `console.log('position received', payload)`
- [ ] In `GlobeView`, after `GLOBE_READY` is received, call `bridgeController.send(OutboundMessage.updatePosition, { 'lat': 51.5, 'lon': -0.1, 'altKm': 420 })` after a 3-second delay
- [ ] Verify in browser console (remote debugging) that the position is received in JS
- [ ] Write unit test for `BridgeController.send` verifying JSON serialization is correct
