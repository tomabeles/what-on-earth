# WOE-018: DeviceOrientation Model and orientationStream

**ID:** WOE-018
**Phase:** 3 — Sensor Fusion and AR Camera
**Step:** 3.2
**Depends on:** WOE-017

---

## Description

Finalize the `DeviceOrientation` model (TECH_SPEC §9.2) and expose `SensorFusionEngine.orientationStream` as the public API consumed by the bridge and any other subsystem that needs device orientation.

**Reference:** TECH_SPEC §7.2, §9.2

---

## Definition of Done

- [ ] `DeviceOrientation` class matches TECH_SPEC §9.2 exactly: `headingDeg`, `pitchDeg`, `rollDeg`, `reliable`, `timestamp`
- [ ] `SensorFusionEngine` exposes `Stream<DeviceOrientation> get orientationStream`
- [ ] Stream is a broadcast stream (multiple listeners supported)
- [ ] `OrientationSample` is renamed or converted to `DeviceOrientation` (unified naming)
- [ ] Unit test: `orientationStream` emits after engine start and stops emitting after engine stop

---

## Sub-tasks

- [ ] Finalize `DeviceOrientation` as an immutable class with `toJson()` for bridge serialization
- [ ] Rename `OrientationSample` to `DeviceOrientation` everywhere (or map between them at the isolate boundary)
- [ ] Expose `Stream<DeviceOrientation> get orientationStream` as a public getter on `SensorFusionEngine`
- [ ] Create a `sensorFusionEngineProvider` Riverpod provider that holds the engine instance
- [ ] Create a `orientationStreamProvider` `StreamProvider` that exposes `engine.orientationStream`
- [ ] Widget test: verify a `ref.watch(orientationStreamProvider)` widget rebuilds when new orientation arrives
